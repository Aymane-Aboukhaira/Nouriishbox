| Muted gray | `#6B6B6B` | Secondary text |

### Typography
- Headings: `DM Serif Display` — editorial, premium feel
- Body: `Nunito` — round, legible, warm

### Shape rules
- No sharp corners anywhere — `rounded-2xl` (24px) or `rounded-3xl` (32px) on all cards
- Soft floating shadows: `shadow-[0_12px_40px_rgba(45,45,45,0.06)]`
- Buttons: `whileHover={{ scale: 1.02 }}` + `whileTap={{ scale: 0.98 }}` on all interactive elements

### UX principles
1. Anticipate needs before the user clicks
2. Context over raw numbers — "44g to go" not "120g / 164g"
3. Motion communicates state — not just decoration
4. One primary action per screen
5. Personalisation that's visible — goal-aware copy throughout

## 4. Zustand stores (`lib/store.ts`)

All 9 stores use `persist` middleware. Store 9 (`useAuthStore`) uses cookie storage for middleware compatibility. All others use localStorage.

### Store 1 — `useProfileStore`
```ts
interface UserProfile {
  name: string
  age: number
  gender: 'male' | 'female'
  weight: number        // kg
  height: number        // cm
  activityLevel: 'sedentary' | 'light' | 'moderate' | 'active' | 'very_active'
  goal: 'lose_weight' | 'build_muscle' | 'maintain' | 'eat_cleaner'
  dailyCalories: number // calculated via Mifflin-St Jeor
  protein: number       // g — 30% of calories
  carbs: number         // g — 45% of calories
  fats: number          // g — 25% of calories
}
```

### Store 2 — `useMealsStore`
- 30+ mock meals with: `id`, `name`, `kcal`, `protein`, `carbs`, `fats`, `category`, `allergens[]`, `isVegan`, `isNew`, `rating?`
- Actions: `addMeal`, `editMeal`, `deleteMeal`

### Store 3 — `usePlannerStore`
- 7-day meal plan: array of `{ dayKey: string, meal: Meal | null, isPaused: boolean, consumed: boolean }`
- Day keys: `'monday'` through `'sunday'`
- Actions: `assignMeal(dayKey, mealId)`, `removeMeal(dayKey)`, `togglePause(dayKey)`, `setConfirmed(confirmed)`, `updateMealStatus(dayKey, mealId, consumed)`
- `isConfirmed: boolean`
- `pausedDays: string[]`

### Store 4 — `useFamilyStore`
- Array of family member profiles with individual macro targets
- Actions: `addMember`, `editMember`, `removeMember`, `assignMealToMember`

### Store 5 — `useOrdersStore`
- Admin Kanban: orders with stages `'pending' | 'preparing' | 'out_for_delivery' | 'delivered'`
- Actions: `updateOrderStage(orderId, stage)`

### Store 6 — `useSubscriptionStore`
```ts
interface SubscriptionState {
  status: 'active' | 'paused' | 'cancelled' | null
  plan: 'solo' | 'couple' | 'family' | null
  orderId: string | null
  startDate: string | null  // ISO string
}
// Action added in checkout build:
setSubscription(data: { plan, status, orderId, startDate }): void
```

### Store 7 — `usePointsStore`
- `points: number` — NourishPoints balance
- `streak: number` — consecutive days with logged meals
- Actions: `addPoints(amount)`, `incrementStreak()`, `resetStreak()`

### Store 8 — `useClinicStore`
- `messages: { id, sender: 'user' | 'ai', text, timestamp }[]`
- Actions: `addMessage`, `clearHistory`

### Store 9 — `useAuthStore` (cookie storage)
```ts
interface AuthUser {
  id: string        // crypto.randomUUID()
  name: string
  email: string
  phone: string
  address: string
  createdAt: string // ISO string
}
interface AuthState {
  user: AuthUser | null
  isAuthenticated: boolean
  signUp(data): void    // creates user, sets isAuthenticated true
  signIn(email): boolean // mock — checks email match
  signOut(): void
}
```
Cookie storage implementation allows Next.js middleware to read auth state server-side.

## 6. Route protection

`middleware.ts` at project root:
- Protects `/client/*` and `/admin/*`
- Reads `auth-storage` cookie (written by `useAuthStore` cookie storage)
- Redirects unauthenticated users to `/auth/signin?from=[originalPath]`
- Sign-in page reads `from` param and redirects back after successful login

## 11. Nutrition engine

Used in the onboarding wizard and profile editor:

```
BMR (male)   = (10 × weight_kg) + (6.25 × height_cm) - (5 × age) + 5
BMR (female) = (10 × weight_kg) + (6.25 × height_cm) - (5 × age) - 161

Activity multipliers:
  sedentary   = 1.2
  light       = 1.375
  moderate    = 1.55
  active      = 1.725
  very_active = 1.9

TDEE = BMR × activity_multiplier

Goal adjustments:
  lose_weight  = TDEE - 500
  build_muscle = TDEE + 300
  maintain     = TDEE
  eat_cleaner  = TDEE

Macro split:
  protein = (dailyCalories × 0.30) / 4   → grams
  carbs   = (dailyCalories × 0.45) / 4   → grams
  fats    = (dailyCalories × 0.25) / 9   → grams
```

## 16. Key technical patterns used throughout

### `hasMounted` guard (used everywhere with Zustand)
```tsx
const [hasMounted, setHasMounted] = useState(false)
useEffect(() => setHasMounted(true), [])
if (!hasMounted) return <SkeletonComponent />
// Now safe to read Zustand stores
```

### Faux-viewport modal (never `position: fixed`)
```tsx
<div style={{
  minHeight: '500px',
  background: 'rgba(45,45,45,0.5)',
  display: 'flex',
  alignItems: 'center',
  justifyContent: 'center',
  borderRadius: '24px'
}}>
  <div className="modal-card">...</div>
</div>
```

### Sticky drawer (never `position: fixed`)
```tsx
<div style={{ position: 'sticky', bottom: 0 }}>
  {/* Always last child of page content flow, outside any overflow:hidden parent */}
</div>
```

### dnd-kit + Framer Motion separation rule
```tsx
// CORRECT — motion.div is visual only, listeners on plain div
<motion.div animate={{ opacity: isDragging ? 0.5 : 1 }}>
  <div ref={setNodeRef} style={...} {...attributes} {...listeners}>
```

## 17. Prompt style guide for continuation

When generating prompts for Antigravity (the AI coding agent):

1. Always start with the full context block (stack, design tokens, what's already built)
2. State the philosophy/intention before the spec
3. Use `"Complete replacement of [file]"` or `"Targeted edit to [file]"` — be explicit
4. Provide TypeScript interfaces for new data structures
5. Provide algorithm logic in code blocks — don't leave it to interpretation
6. Specify animation values precisely: `y: 40 → 0`, `opacity: 0 → 1`, `duration: 400ms`

8. Always specify the `position: sticky` pattern for drawers and the faux-viewport pattern for modals

## 18. What NOT to do (lessons learned)

- Never use `position: fixed` on drawers or modals — collapses iframe viewport to `min-height: 100px`
- Never place `motion.div` with `...listeners` from `useSortable` — breaks dnd-kit entirely
- Never de-duplicate meals across planner days — users legitimately repeat meals
- Never hardcode a day index (0) when assigning meals — always use the `dayKey` string
- Never use `position: fixed` on the confetti burst — use `position: absolute` inside a `position: relative` wrapper
- Never animate the modal/drawer container height with `height: 'auto'` without `overflow: hidden` on the `motion.div`
- Never skip the `hasMounted` guard — Zustand localStorage reads cause SSR hydration mismatches in Next.js 14 App Router
- Never wrap `useSearchParams()` without a `Suspense` boundary in Next.js 14
